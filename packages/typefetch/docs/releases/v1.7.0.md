# TypeFetch Runtime Instrumentation & Overrides

This release adds a small, **additive** instrumentation layer to the client. It
exposes structured request lifecycle events (with the **parsed** input and
**parsed** output) and an optional per-request **override** mechanism that can
force a mock, simulate an error/latency, or swap the request/response validation
schema at runtime ‚Äî without ever touching the contract.

It also stamps **stable metadata** onto every generated endpoint method
(`endpointId` + the original `endpoint` definition), so higher-level tooling can
key cache/events and read a contract's schemas.

This is the foundation for the upcoming framework-agnostic query layer and a
devtools/inspector, but it is useful on its own.

Fully backward compatible: when no instrumentation hook is registered, request
handling is **byte-for-byte identical** to before ‚Äî zero cost, zero behavior
change.

## Endpoint metadata

Each method returned by `client.modules` is still callable exactly as before,
but now carries two read-only properties:

```ts
client.init();
const api = client.modules;

api.user.getUser.endpointId; // "user.getUser" ‚Äî stable "module.endpoint" id
api.user.getUser.endpoint;   // the original contract def (schemas, method, path, mockData, errors)
```

`endpointId` is the natural, collision-free key for a query cache or an event
log. `endpoint` lets tooling read the Zod `request`/`response` schemas without a
separate reference to the contracts object.

## Instrumentation

Register a hook with `client.instrument(...)`. It returns an unsubscribe
function. Multiple hooks can be attached.

```ts
const stop = client.instrument({
  on(event) {
    // event.type is "start" | "success" | "error"
    console.log(event);
  },
});

// later
stop();
```

### Events

Unlike middleware (which only sees the raw `Response`), instrumentation events
carry the **validated** input and the **parsed/typed** output, plus timing and
the endpoint identity:

```ts
type RequestEvent =
  | { type: "start";   requestId: string; endpointId: string; method: Method; url: string; input: unknown; timestamp: number }
  | { type: "success"; requestId: string; endpointId: string; data: unknown; durationMs: number; fromMock: boolean }
  | { type: "error";   requestId: string; endpointId: string; status?: number; error: ErrorLike; durationMs: number };
```

`requestId` correlates the `start` event with its matching `success`/`error`.
`fromMock` is `true` when the result came from mock data (configured or forced).

## Runtime overrides

A hook can also resolve a per-request `Override`, letting tooling change what a
single request does **without mutating the contract**. The first hook to return
an override wins for that request.

```ts
client.instrument({
  resolveOverride(endpointId, input) {
    if (endpointId === "user.getUser") {
      return { mock: { id: "forced", name: "Forced User" } };
    }
    return undefined;
  },
});
```

```ts
type Override = {
  mock?: unknown | ((input: unknown) => unknown); // force mock, bypass network
  error?: { status?: number; code?: string; message?: string; body?: unknown }; // simulate a failure
  latencyMs?: number;                              // inject latency before resolving
  request?: z.ZodTypeAny;                          // swap the request schema at runtime
  response?: z.ZodTypeAny;                         // swap the response schema at runtime
};
```

Semantics:

* **`mock`** ‚Äî bypasses the network regardless of the client's mock mode. A
  function receives the parsed input. The result is still validated against the
  (possibly overridden) response schema, just like configured mock data.
* **`error`** ‚Äî throws a `RichError` and fires the client's `onError` handler,
  exactly like a real failing endpoint, so error paths can be exercised on demand.
* **`latencyMs`** ‚Äî awaited before the request resolves (any path).
* **`request` / `response`** ‚Äî swap the Zod validation schema for this request
  only. The original contract object is never mutated; a shallow copy carries the
  overridden schema. This makes it possible to test structural changes at runtime.

Fields are independent and compose: a swapped `response` schema together with a
forced `mock` validates the mock against the new shape.

## API additions

| Export             | Kind  | Purpose                                                              |
| ------------------ | ----- | ------------------------------------------------------------------- |
| `EndpointMethod<E>` | type  | A callable endpoint method plus `endpointId` / `endpoint` metadata. |
| `RequestEvent`     | type  | The `start` / `success` / `error` lifecycle event union.            |
| `Override`         | type  | Per-request runtime override (mock/error/latency/schema).           |
| `Instrumentation`  | type  | `{ on?, resolveOverride? }` hook shape.                             |
| `client.instrument(hook)` | method | Register a hook; returns an unsubscribe function.            |
| `method.endpointId` | field | Stable `"module.endpoint"` identifier on each generated method.     |
| `method.endpoint`  | field | The original contract definition on each generated method.          |

## Backward compatibility

* `instrument` is opt-in. With no hook registered, the request path is unchanged
  and adds no work.
* `EndpointMethods` return types and call signatures are unchanged; the metadata
  is purely additive on the function value.
* Middleware, retries, mock mode, response wrappers, and typed errors all behave
  as before. Overrides layer on top without altering those flows.

## Design intent

Instrumentation is the **single** extension point that the coming
framework-agnostic query engine and devtools bridge build on:

* the query layer keys its cache by `endpointId` + the request input;
* the devtools bridge attaches one hook to render a live request timeline and to
  serve runtime overrides (force mock, simulate errors/latency, swap schemas).

Keeping this seam thin and additive means those packages never need to fork or
re-implement the request pipeline.

## Future (not in this release)

* An HTTP status on the `success` event for richer timelines.
* A resolved (params-substituted) URL on the `start` event.
* A response transform in `Override` for partial response mutation.
