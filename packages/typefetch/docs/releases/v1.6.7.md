# TypeFetch Typed Error Responses

This release adds **typed per-endpoint error responses**. An endpoint can now
declare the body of its error responses (404, 409, 422, …) alongside its
existing success `response`, so the client can parse and type a failed request's
body, and external tools can document and validate those errors.

Fully backward compatible: endpoints without `errors` behave exactly as before,
byte for byte on the success path.

## Declaring errors

Add an optional `errors` map keyed by HTTP status code. Each value is a plain
Zod schema.

```ts
createUser: {
  method: "POST",
  path: "/users",
  request: z.object({ body: z.object({ email: z.string().email() }) }),
  response: z.object({ id: z.string() }),
  errors: {
    409: z.object({ code: z.literal("EMAIL_TAKEN"), conflictField: z.string() }),
    422: z.object({ code: z.literal("INVALID"), issues: z.array(z.string()) }),
  },
}
```

* Key by HTTP status **code** (number) — maps 1:1 to OpenAPI `responses` and to
  `RichError.status`.
* If a single status can carry multiple bodies, the schema value itself can be a
  `z.discriminatedUnion("code", [...])` — no need to complicate the map.
* `response` stays the success (2xx) entry; `errors` is added alongside it. The
  two are never merged.

## Client behavior

On a failed request, the client looks up `endpoint.errors[res.status]`. If a
schema is declared, it `safeParse`s the body and attaches the parsed value to a
new `RichError.data` field, recording whether it validated in
`RichError.dataParsed`.

Parsing is **fail-open** — it never throws and never masks the real error:

* No schema for the status → `RichError.data` is the raw JSON body,
  `RichError.dataParsed` is `false`.
* Body fails validation → `RichError.data` is the raw JSON body,
  `RichError.dataParsed` is `false`.
* Schema declared and body passes → `RichError.data` is the parsed, typed body,
  `RichError.dataParsed` is `true`.
* All existing `RichError` fields (`message`, `status`, `code`, `title`,
  `detail`, `errors`) are unchanged.

The wrapper (`setResponseWrapper`) and non-wrapper flows both build the error
from the same JSON body, so error-body parsing stays consistent across them.

## Typed guard

`catch` clauses are `unknown`, so control flow alone can't narrow a thrown
error. `isContractError` references the endpoint to recover the body type from
the status literal:

```ts
import { isContractError } from "@tahanabavi/typefetch";
import { contracts } from "./contracts";

try {
  await api.user.createUser({ body: { email } });
} catch (e) {
  if (isContractError(contracts.user.createUser, e, 409)) {
    e.data.conflictField; // fully typed from the 409 schema
  }
}
```

It returns `true` only when `e` is a `RichError` whose `status` matches **and**
whose body actually validated against the declared schema
(`RichError.dataParsed === true`); when it does, `e.data` is narrowed to that
status's declared body. If the server returns that status with a body that
doesn't match the contract, the guard returns `false` rather than narrowing to a
type the runtime value doesn't have.

## Type helpers

```ts
import type { InferError, InferErrors } from "@tahanabavi/typefetch";

type Conflict = InferError<typeof contracts.user.createUser, 409>;
// { code: "EMAIL_TAKEN"; conflictField: string }

type AllErrors = InferErrors<typeof contracts.user.createUser>;
// { 409: {...}; 422: {...} }
```

`InferErrors<E>` resolves to `{}` and `InferError<E, S>` to `never` for
endpoints (or statuses) with no declared error schema.

## API additions

| Export                       | Kind  | Purpose                                                    |
| ---------------------------- | ----- | ---------------------------------------------------------- |
| `ErrorResponsesMap`          | type  | `Record<number, z.ZodTypeAny>` — the shape of `errors`.    |
| `EndpointDef<TReq, TRes, TErr>` | type  | New optional third generic; adds the `errors?` field.   |
| `InferErrors<E>`             | type  | `{ [status]: body }` for an endpoint.                      |
| `InferError<E, S>`           | type  | A single status's error body type.                         |
| `RichError.data`             | field | Parsed (or raw, on fail-open) error body.                  |
| `RichError.dataParsed`       | field | `true` only when `data` validated against the schema.      |
| `isContractError(ep, e, s)`  | fn    | Typed guard narrowing a caught error by status.            |

## Consuming from external tools

`errors` is a plain `Record<number, z.ZodTypeAny>` — each value is a real Zod
schema, not a wrapper object — so tools such as
[`@tahanabavi/typefetch-nestjs`](https://github.com/tahanabavi/typefetch-nestjs)
can iterate `endpoint.errors` to emit an OpenAPI `responses[status]` entry per
declared error and validate that server error bodies match the contract.

## Backward compatibility

* `errors` is optional; omitting it is today's behavior, unchanged.
* `response` semantics and `EndpointMethods` return types are unchanged — errors
  are thrown, not returned. The endpoint method still resolves to the success
  type.
* Zod 4 remains the peer; `errors` values use `z.ZodTypeAny` as elsewhere.

## Future (not in this release)

* Per-status mock/example bodies for error responses (would feed OpenAPI
  `examples`).
* A client-level `matchError(endpoint, error)` returning a discriminated
  `{ status, data }` union.
