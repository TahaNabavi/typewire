# TypeFetch CLI Feature

This folder adds a CLI layer on top of the TypeFetch testing runner.

## Commands

### Create required files

```bash
typefetch init
```

Creates:

```txt
typefetch.test.config.ts
typefetch.env.example
typefetch-report/.gitkeep
test-fixtures/.gitkeep
```

Useful options:

```bash
typefetch init --force
typefetch init --package @tahanabavi/typefetch
typefetch init --contracts-path ./src/contracts
typefetch init --output .
```

### Run tests

```bash
typefetch test
```

Common usage:

```bash
typefetch test \
  --config ./typefetch.test.config.ts \
  --mode full \
  --base-url http://localhost:3000 \
  --format markdown,json,html \
  --output ./typefetch-report/report
```

The CLI exits with code `1` when at least one test fails, so it works in CI.

### List endpoints

```bash
typefetch list
```

Prints all endpoints discovered from the config contracts.

## Generated config example

```ts
import { ApiClient, defineTypeFetchTestConfig } from "@tahanabavi/typefetch";
import { contracts } from "./src/contracts";

export default defineTypeFetchTestConfig({
  contracts,

  createClient: ({ baseUrl, token }) => {
    const client = new ApiClient(
      {
        baseUrl: baseUrl ?? process.env.API_BASE_URL ?? "http://localhost:3000",
        tokenProvider: async () => token ?? process.env.API_TOKEN ?? "",
      },
      contracts,
    );

    client.init();
    return client;
  },

  options: {
    mode: "full",
    timeout: 10_000,
    includeDestructive: false,
    stopOnFail: false,
  },

  report: {
    output: "./typefetch-report/report",
    formats: ["markdown", "json", "html"],
  },
});
```

# TypeFetch Testing Feature

Contract-driven API testing, auto-generated requests, and report generation for TypeFetch.

## Overview

This update adds a new optional testing layer to TypeFetch. The goal is to test every endpoint defined in your contracts without manually filling forms in a UI. The test runner discovers endpoints from the contract, creates or receives valid inputs, executes the endpoint through the existing `ApiClient`, validates the response with the endpoint Zod schema, and generates a useful report in Markdown, HTML, or JSON.

The feature is designed as a separate `testing/` module, not as extra logic inside the core `ApiClient`. This keeps the client focused on request execution while allowing testing, reporting, fixtures, and CLI behavior to evolve independently.

```txt
Zod Contracts
     ↓
Endpoint discovery
     ↓
Auto input generator + custom test cases
     ↓
ApiClient execution
     ↓
Response validation
     ↓
Markdown / HTML / JSON report
```

## Why this feature exists

When a project grows, adding endpoints becomes easy but testing them manually becomes slow. This update solves that by making your TypeFetch contracts the source of truth for API testing.

Instead of writing a separate test for every endpoint from scratch, you can define lightweight metadata directly on the endpoint:

- which endpoints should be tested
- which input should be used
- whether an endpoint is destructive
- which tags it belongs to
- what status code is expected
- how to store data for later endpoint tests
- which custom assertion should run after the response

## What changed

### 1. New endpoint `test` metadata

`EndpointDef` now supports an optional `test` field:

```ts
export type EndpointDef<
  TReq extends RequestSchema,
  TRes extends ResponseSchema,
> = {
  method: Method;
  path: string;
  auth?: boolean;
  request: TReq;
  response: TRes;
  mockData?: (() => z.infer<TRes>) | z.infer<TRes>;
  encryption?: EncryptionConfig<z.infer<TReq>, z.infer<TRes>>;
  headers?:
    | Record<string, string>
    | ((input: z.infer<TReq>) => Record<string, string>);
  bodyType?: "json" | "form-data";

  test?: EndpointTestConfig<z.infer<TReq>, z.infer<TRes>>;
};
```

## Core concepts

## ApiTestRunner

The runner discovers all endpoints from the contract and runs one or more phases depending on the selected mode.

```ts
const report = await createApiTestRunner({
  client,
  contracts,
  options: {
    mode: "full",
    timeout: 10_000,
    includeDestructive: false,
  },
}).run();
```

## Test modes

```ts
export type ApiTestMode = "schema" | "mock" | "live" | "full";
```

### `schema`

Generates or receives input and validates it against the request schema. No network request is sent.

Useful for:

- checking that auto-generated input is valid
- checking custom test inputs
- running fast CI validation

### `mock`

Runs against endpoint `mockData` when available and validates the mock response with the endpoint response schema.

Useful for:

- validating that mock data still matches contracts
- testing UI or API clients without backend availability

### `live`

Runs the endpoint through `ApiClient` and validates the real response.

Useful for:

- smoke testing the real API
- staging environment checks
- contract-backend compatibility checks

### `full`

Runs all useful phases:

1. schema
2. mock, if mock data exists
3. live

## Runner options

```ts
export type ApiTestRunnerOptions = {
  mode?: ApiTestMode;
  timeout?: number;
  concurrency?: number;
  stopOnFail?: boolean;
  includeDestructive?: boolean;
  includeTags?: string[];
  excludeTags?: string[];
  autoInput?: AutoInputOptions;
  requestOptions?: RequestOptions;
};
```

### `mode`

Selects how endpoints should be tested.

Default:

```ts
mode: "live";
```

### `timeout`

Per-request timeout in milliseconds.

Default:

```ts
timeout: 10_000;
```

### `concurrency`

Reserved for parallel endpoint execution.

Recommended default for now:

```ts
concurrency: 1;
```

Sequential execution is safer because many endpoint tests depend on context values created by previous endpoints.

### `stopOnFail`

Stops the runner after the first failed endpoint.

```ts
stopOnFail: true;
```

### `includeDestructive`

Controls whether destructive endpoints are executed.

Default:

```ts
includeDestructive: false;
```

Endpoints marked as destructive are skipped unless this is enabled.

### `includeTags`

Runs only endpoints that match one of the selected tags.

```ts
includeTags: ["smoke", "user"];
```

### `excludeTags`

Skips endpoints that match one of the excluded tags.

```ts
excludeTags: ["danger", "payment"];
```

### `autoInput`

Controls the automatic input generator.

```ts
autoInput: {
  includeOptional: true,
  includeArrayItems: true,
  values: {
    "path.id": "user-1",
    email: "test@example.com",
  },
}
```

## Endpoint test config

```ts
export type EndpointTestConfig<TReq, TRes> = {
  enabled?: boolean;
  tags?: string[];
  destructive?: boolean;
  input?: TReq | EndpointTestInputFactory<TReq>;
  cases?: EndpointTestCase<TReq, TRes>[];
  setup?: (ctx: EndpointTestContext) => void | Promise<void>;
  teardown?: (ctx: EndpointTestContext) => void | Promise<void>;
};
```

## Basic endpoint example

```ts
import { z } from "zod";
import type { Contracts } from "./types";

export const contracts = {
  user: {
    listUsers: {
      method: "GET",
      path: "/users",
      request: z.object({
        query: z
          .object({
            page: z.number().optional(),
            active: z.boolean().optional(),
          })
          .optional(),
      }),
      response: z.array(
        z.object({
          id: z.string(),
          name: z.string(),
          email: z.string().email(),
        }),
      ),
      test: {
        tags: ["user", "smoke"],
        input: {
          query: {
            page: 1,
            active: true,
          },
        },
      },
    },
  },
} satisfies Contracts;
```

## Endpoint with context dependency

Use context when one endpoint depends on the result of another endpoint.

```ts
const userResponseSchema = z.object({
  id: z.string(),
  name: z.string(),
  email: z.string().email(),
});

type UserResponse = z.infer<typeof userResponseSchema>;

export const contracts = {
  user: {
    createUser: {
      method: "POST",
      path: "/users",
      request: z.object({
        body: z.object({
          name: z.string().min(3),
          email: z.string().email(),
        }),
      }),
      response: userResponseSchema,
      test: {
        tags: ["user", "write"],
        cases: [
          {
            name: "create valid user",
            input: {
              body: {
                name: "Taha Nabavi",
                email: "taha@example.com",
              },
            },
            expect: ({ response, ctx }) => {
              const createdUser = response as UserResponse;
              ctx.set("createdUserId", createdUser.id);
            },
          },
        ],
      },
    },

    getUserById: {
      method: "GET",
      path: "/users/:id",
      request: z.object({
        path: z.object({
          id: z.string(),
        }),
      }),
      response: userResponseSchema,
      test: {
        tags: ["user", "smoke"],
        input: (ctx) => ({
          path: {
            id: (ctx.get("createdUserId") as string | undefined) ?? "user-1",
          },
        }),
      },
    },
  },
} satisfies Contracts;
```

## Expected error example

Sometimes an endpoint should return an error. For example, a missing user should return `404`.

```ts
notFoundExpected: {
  method: "GET",
  path: "/users/missing",
  request: z.object({}),
  response: z.object({
    id: z.string(),
    name: z.string(),
    email: z.string().email(),
  }),
  test: {
    tags: ["user", "negative"],
    cases: [
      {
        name: "404 is expected",
        input: {},
        expectStatus: 404,
      },
    ],
  },
}
```

When `expectStatus` matches the thrown error status, the result is marked as passed.

## Destructive endpoint example

```ts
deleteUser: {
  method: "DELETE",
  path: "/users/:id",
  request: z.object({
    path: z.object({
      id: z.string(),
    }),
  }),
  response: z.object({
    deleted: z.boolean(),
  }),
  test: {
    tags: ["user", "danger"],
    destructive: true,
    input: (ctx) => ({
      path: {
        id: (ctx.get("createdUserId") as string | undefined) ?? "user-1",
      },
    }),
  },
}
```

This endpoint is skipped by default.

To run destructive endpoints:

```ts
const report = await createApiTestRunner({
  client,
  contracts,
  options: {
    includeDestructive: true,
  },
}).run();
```

## Disabled endpoint example

```ts
disabledEndpoint: {
  method: "GET",
  path: "/disabled",
  request: z.object({}),
  response: z.object({ ok: z.boolean() }),
  test: {
    enabled: false,
  },
}
```

The runner will include it in the report as skipped.

## Form-data / upload example

Your client already supports `bodyType: "form-data"`. The testing feature can generate or receive form-data compatible input.

```ts
uploadAvatar: {
  method: "POST",
  path: "/users/:id/avatar",
  bodyType: "form-data",
  request: z.object({
    path: z.object({
      id: z.string(),
    }),
    body: z.object({
      file: z.any(),
      alt: z.string().optional(),
    }),
  }),
  response: z.object({
    uploaded: z.boolean(),
  }),
  test: {
    tags: ["user", "upload"],
    input: {
      path: {
        id: "user-1",
      },
      body: {
        file: new Blob(["fake-file-content"], { type: "text/plain" }),
        alt: "Avatar",
      },
    },
  },
}
```

For Node/Jest environments where `Blob` or `File` is not available, use `fileFactory`:

```ts
const report = await createApiTestRunner({
  client,
  contracts,
  options: {
    autoInput: {
      fileFactory: () => "fake-file-content",
    },
  },
}).run();
```

## Running the test runner

```ts
import { ApiClient } from "./client";
import { createApiTestRunner } from "./testing";
import { writeReportFiles } from "./testing/node-reporter";
import { contracts } from "./contracts";

async function main() {
  const client = new ApiClient(
    {
      baseUrl: "http://localhost:3000",
      tokenProvider: async () => process.env.ACCESS_TOKEN ?? "",
    },
    contracts,
  );

  client.init();

  const report = await createApiTestRunner({
    client,
    contracts,
    context: {
      userId: "user-1",
    },
    options: {
      mode: "full",
      timeout: 10_000,
      includeDestructive: false,
      excludeTags: ["payment", "danger"],
    },
  }).run();

  await writeReportFiles(report, "./reports/typefetch-report.md");
  await writeReportFiles(report, "./reports/typefetch-report.html");
  await writeReportFiles(report, "./reports/typefetch-report.json");
}

main().catch((error) => {
  console.error(error);
  process.exit(1);
});
```

## Auto input generator

The input generator is a best-effort tool. It can create values for common Zod schemas.

Supported common types:

- `z.string()`
- `z.string().min(n)`
- `z.string().email()`
- `z.string().url()`
- `z.number()`
- `z.boolean()`
- `z.date()`
- `z.array(...)`
- `z.object(...)`
- `z.enum(...)`
- `z.union(...)`
- `z.optional(...)`
- `z.nullable(...)`
- `z.default(...)`
- `z.record(...)`

Example:

```ts
const schema = z.object({
  path: z.object({
    id: z.string(),
  }),
  body: z.object({
    name: z.string().min(12),
    email: z.string().email(),
  }),
});

const input = generateInput(schema);

schema.parse(input); // valid
```

The latest generator fix ensures semantic defaults like `"Test Name"` are padded when the schema requires a longer minimum length.

## Auto input overrides

You can override generated values by full path, partial path, or field name.

```ts
const input = generateInput(schema, {
  values: {
    "body.name": "Taha Nabavi",
    "body.email": "taha@example.com",
    id: "user-1",
  },
});
```

## Recommended testing strategy

Use three levels of testing:

### 1. Schema mode in every commit

Fast and safe.

```bash
npm run test:contracts
```

### 2. Mock mode in frontend/client CI

Makes sure mock responses still match response contracts.

### 3. Live mode against staging

Run this before merging backend/API changes.

Never run destructive tests against production.

## Best practices

### Keep tests close to contracts

Endpoint-specific test metadata should live next to the endpoint definition. This makes the contract self-documenting.

### Use tags aggressively

Good tag examples:

```ts
tags: ["user", "smoke"];
tags: ["auth", "negative"];
tags: ["upload"];
tags: ["danger"];
tags: ["payment"];
```

### Mark dangerous endpoints

Use:

```ts
destructive: true;
```

for endpoints that delete, reset, charge money, send external emails, mutate production-like data, or trigger irreversible actions.

### Prefer explicit input for real APIs

Auto input is helpful, but real APIs often need real IDs and valid fixture data. Use `test.input` or `test.cases` for important endpoints.

### Use context for flows

For dependent flows, use `ctx.set` and `ctx.get`.

```ts
ctx.set("userId", response.id);
ctx.get("userId");
```

### Do not over-test implementation details

Reports can include different valid skip reasons depending on runner order. Tests should verify the important behavior, not fragile internal reason strings.

### Before

```ts
  }),

```

### After

```ts
getUserById: {
  method: "GET",
  path: "/users/:id",
  request: z.object({

  }),
  response: userResponseSchema,

    tags: ["user", "smoke"],

        id: "user-1",
      },
    },
  },
}
```

No existing endpoint has to change immediately because `test` is optional.

## Known limitations

### Auto-generated values are not business-aware

The generator can satisfy the Zod schema, but it cannot know business rules. For example, it can generate a valid string ID, but it cannot know whether that ID exists in your database.

Use explicit inputs or context for business-dependent endpoints.

### Flow ordering matters

If `getUserById` depends on `createUser`, make sure `createUser` is discovered and executed first or provide initial context.

Future versions can add an explicit dependency graph.

### Concurrency should stay low for dependent tests

For now, use:

```ts
concurrency: 1;
```

when tests share context.

### Production safety

Do not enable `includeDestructive` against production environments.

Keep this feature as a separate module:

```txt

```

and export it from the package as:

